## 一、系统概述

### 1.1 产品定位

明眸（VisionMind）是基于人脸表情识别的校园情绪/投入度分析平台。通过 AI 摄像头采集学生在课堂场景中的面部表情，实时分析七类基本情绪概率分布，映射为三种学习状态（投入积极/困惑思考/疲惫退缩），为教师、年级主任、心理教师、校领导和家长提供差异化的情绪洞察和关怀支持。

### 1.2 核心价值

- **教师**: 课堂复盘（情绪时间轴）、个体关怀跟进
- **年级主任**: 年级情绪趋势、跨班对比
- **心理教师**: 全校情绪雷达、个体深描档案
- **校领导**: 系统运行总览、跨年级分析
- **家长**: 子女情绪概览（脱敏）

### 1.3 设计原则

1. **隐晦关怀优先**: 所有预警和关怀操作避免标签化，保护学生心理安全
2. **场景化隐私分级**: 不同摄像头场景采集不同分析深度
3. **AI 辅助而非替代**: AI 输出为辅助决策参考，教师保留最终判断权
4. **情绪判定一致性**: 全平台共用统一情绪映射函数，三态判定规则唯一定义

## 二、情绪模型

### 2.1 Ekman 七类原始情绪

标准顺序与标签（全平台统一）：

| 序号 | key | 中文标签 | 颜色 | 效价 |
|------|-----|---------|------|------|
| 0 | happy | 快乐 | #F4B942 | 正向 |
| 1 | neutral | 中性 | #9BAFBC | 中性 |
| 2 | surprise | 惊讶 | #9B9ECE | 中性（认知信号） |
| 3 | sad | 悲伤 | #6B8CAE | 负向 |
| 4 | angry | 愤怒 | #E07C7C | 负向 |
| 5 | disgust | 厌恶 | #8DAF8D | 负向 |
| 6 | fear | 恐惧 | #C9A0B8 | 负向 |

**实现位置**: `emotionModel.ts` → `RAW_EMOTIONS` 常量

### 2.2 三态学习状态

| 状态 | 英文 | 颜色 | 标签 |
|------|------|------|------|
| 投入积极 | ENGAGED | #F2C94C | 奖励激活，认知资源充分投入 |
| 困惑思考 | CONFUSED | #9B9ECE | 认知冲突，信息加工中 |
| 疲惫退缩 | WITHDRAWN | #BDBDBD | 情绪调节资源耗尽或低唤醒 |
| 未识别 | UNKNOWN | transparent | 采集不足或置信度过低 |

### 2.3 统一映射函数 `scoresToState`

**唯一实现位置**: `emotionModel.ts`

**判定优先级**（从上到下，命中即返回）:

```
1. 快乐 ≥ 33% → ENGAGED
   理论依据: 多巴胺奖励激活，Pekrun 控制-价值理论的"享受"情绪

2. 惊讶 ≥ 18% → CONFUSED
   理论依据: Piaget 认知冲突信号，认知失调优先于负向判定

3. 负向累积(悲伤+愤怒+恐惧+厌恶) ≥ 25% 且 快乐 < 15% 且 惊讶 < 12% → WITHDRAWN
   理论依据: Gross 情绪调节资源耗竭，排除认知冲突后的纯负向脱离

4. 中性 ≥ 28% 且 快乐 < 22% → CONFUSED
   理论依据: 无强烈情绪信号时的"静默认知加工"状态

5. 兜底 → WITHDRAWN
   理论依据: 低唤醒状态安全兜底，宁可多关注不漏报
```

### 2.4 可视化辅助映射 `EMOTION_STATE_MAP`

每个原始情绪对应的主要学习状态（用于课堂复盘粒子标记的概率条高亮）:

| 情绪 | 映射状态 | 依据 |
|------|---------|------|
| 快乐 | ENGAGED | 正向情感驱动 |
| 中性 | ENGAGED | 平静专注 |
| 惊讶 | CONFUSED | 认知冲突核心信号 |
| 悲伤 | WITHDRAWN | 情绪低落 |
| 愤怒 | WITHDRAWN | 挫败导致脱离 |
| 厌恶 | WITHDRAWN | 排斥反应 |
| 恐惧 | WITHDRAWN | 焦虑阻断学习 |

### 2.5 心理学解读映射 `PSYCH_MAP`

为教师提供的"情绪→学习状态"解读说明，包含判定规则文字描述和教育心理学解释。`PSYCH_MAP.rule` 字段必须与 `scoresToState` 的实际判定逻辑严格一致。

### 2.6 禁止事项

- **禁止**各页面自定义情绪映射函数
- **禁止**各页面定义自己的七类情绪标签或顺序
- **禁止**PSYCH_MAP 描述与 scoresToState 逻辑出现矛盾
- 所有情绪判定必须 import `emotionModel.ts` 的统一函数

## 四、关怀工作流

### 4.1 关怀状态机（完整规格）

```
                        ┌─────────────────────────────────────────────────────┐
                        │                                                     │
                        ▼                                                     │
  ┌──────────┐    AI 预警    ┌──────────┐  老师关注  ┌──────────────┐  记录关怀  │
  │ AI 发现  │ ──────────→  │ PENDING  │ ────────→ │ ACKNOWLEDGED │ ────────→  │
  │ (AtRisk) │              │ 待关注   │           │ 已关注       │            │
  └──────────┘              └──────────┘           └──────────────┘            │
         │                       │                        │                    │
         │ 忽略(dismiss)         │ 7天后未改善            │ 多次关怀无改善       │
         ▼                       ▼                        ▼                    │
  ┌──────────┐            ┌──────────┐            ┌──────────────┐             │
  │ DISMISSED│            │ RE_ALERT │            │  RESOLVED    │             │
  │ 已忽略   │            │ 重新预警  │            │ 已恢复/已转介│             │
  └──────────┘            └──────────┘            └──────────────┘             │
         │                       │                                                   │
         │ 7天回访               └───────────────────────────────────────────────┘
         ▼                    (回到 PENDING 流程)
  ┌──────────┐
  │ RE_ALERT │
  │ 重新预警  │
  └──────────┘
```

### 4.2 状态定义

| 状态 | key | 含义 | 触发方式 |
|------|-----|------|---------|
| 待关注 | PENDING | AI 新发现，等待教师处理 | AI 引擎自动触发 |
| 已关注 | ACKNOWLEDGED | 教师已知悉，开始跟进 | 教师点击"我会关注的" |
| 已忽略 | DISMISSED | 教师判断暂时无需跟进 | 教师点击"暂时忽略" |
| 重新预警 | RE_ALERT | 已忽略但 7 天后仍未改善 | 系统自动重新触发 |
| 已恢复 | RESOLVED_RECOVERED | 学生情绪状态恢复正常 | 教师确认 / AI 自动判定连续 5 天 ENGAGED ≥ 60% |
| 已转介 | RESOLVED_REFERRED | 转介至心理教师或外部机构 | 教师操作 |
| 误报 | RESOLVED_FALSE_ALARM | AI 判定有误 | 教师标注 |

### 4.3 统一 CareItem 接口（待实现）

当前 mockData.ts 和 CareRecordsPage.tsx 各自定义 CareItem 接口，需统一为：

```typescript
interface CareItem {
  uid: string;
  studentId: number;
  name: string;
  gender: 'male' | 'female';
  className: string;
  status: CareStatus;
  daysPending: number;          // PENDING/ACKNOWLEDGED 状态持续天数
  dismissedAt?: string;         // DISMISSED 状态的时间戳
  reAlertAt?: string;           // 预计重新预警时间
  resolvedAt?: string;          // 结案时间
  resolvedType?: ResolvedType;  // 结案类型
  prompt: string;               // AI 生成的简短描述
  detailPrompt: string;         // AI 生成的详细描述
  records: CareRecord[];        // 关怀操作记录
}

type CareStatus = 'PENDING' | 'ACKNOWLEDGED' | 'DISMISSED' | 'RE_ALERT' 
                | 'RESOLVED_RECOVERED' | 'RESOLVED_REFERRED' | 'RESOLVED_FALSE_ALARM';
type ResolvedType = 'RECOVERED' | 'REFERRED' | 'FALSE_ALARM';

interface CareRecord {
  id: string;
  timestamp: string;
  type: 'NOTE' | 'PHONE' | 'MEETING';
  content: string;
  author: string;
}
```

### 4.4 dismiss 回访规则（待实现）

- 教师点击"暂时忽略" → 状态转为 DISMISSED，记录 `dismissedAt`
- 7 天后（`dismissedAt + 7 days`），系统自动检查学生情绪状态
- 若仍处于 WITHDRAWN 或 CONFUSED ≥ 3天/7天 → 重新触发预警，状态转为 RE_ALERT
- RE_ALERT 状态在 UI 上以特殊标记展示，提醒教师"此前已忽略但未改善"

### 4.5 关怀数据源统一（待实现）

DewdropCare 和 CareRecordsPage 必须操作同一数据源。当前两者各自维护独立列表，需改为共享 `CARE_LIST`（mockData.ts 导出），两处的状态变更（acknowledge/dismiss/add note）同步反映到对方 UI。

## 五、学年管理

### 5.1 数据模型

```typescript
interface SemesterData {
  label: string;     // "第一学期" | "第二学期"
  start: string;     // ISO 日期 "2026-09-01"
  end: string;       // ISO 日期 "2027-01-20"
}

interface YearData {
  label: string;     // "2026-2027学年"
  status: 'ACTIVE' | 'ARCHIVED';
  isCurrent: boolean;
  semesters: SemesterData[];
}
```

### 5.2 初始数据

- 2026-2027学年（当前，ACTIVE）
  - 第一学期: 2026-09-01 至 2027-01-20
  - 第二学期: 2027-02-20 至 2027-06-30
- 2025-2026学年（已归档，ARCHIVED）
  - 第一学期: 2025-09-01 至 2026-01-20
  - 第二学期: 2026-02-17 至 2026-06-30

### 5.3 学年过渡流程

1. 管理员在当前学年页面点击"发起学年过渡"
2. 系统在当前学期结束日自动执行:
   - 将当前学年所有数据（学生情绪记录、关怀记录、课表等）归档
   - 创建新学年数据容器（空课表、空情绪记录）
   - 毕业年级学生（初三）标记为"离校"状态
   - 新年级（初一）学生导入初始化
3. 过渡完成后，新学年成为"当前学年"

- Student: 增加 `academicYearLabel`（当前所在学年）
